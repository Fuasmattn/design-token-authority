# TICKET-008: Define and validate a project config schema

**Phase:** 2 — Developer Experience
**Priority:** High
**Effort:** M

## Summary

Introduce a typed, validated project configuration file (`dta.config.ts`) that describes a project's Figma file, token layer structure, brand variants, and desired output targets. This is the central contract that makes the tool whitelabelable across different design systems.

## Background

Today the tool is hardwired: the Figma file key comes from `.env`, output formats from `config.json`, and the collection/layer meaning is implicit in filename patterns. There is no way to describe "my design system has two brands and three screen sizes" without modifying source code.

A config schema externalizes all of this into a per-project file that can live in the consuming repository.

## Acceptance Criteria

- [x] Config file format: `dta.config.ts` (TypeScript, evaluated at runtime)
- [x] Schema defined with `zod` — all fields validated with clear error messages on startup
- [x] Config is loaded by all CLI subcommands via `--config` flag (default: `./dta.config.ts`)
- [x] Config shape covers: Figma connection, layer mapping, brand list, output targets
- [x] A minimal working config is generated by `dta init` (TICKET-015)
- [x] TypeScript types are exported so consuming projects get autocompletion

## Proposed Config Shape

```ts
// dta.config.ts
import { defineConfig } from 'design-token-authority'

export default defineConfig({
  figma: {
    fileKey: process.env.FIGMA_FILE_KEY!,
    personalAccessToken: process.env.FIGMA_PERSONAL_ACCESS_TOKEN!,
  },

  // Map Figma collection names to semantic layer roles
  layers: {
    primitives: 'Primitives(Global)',   // raw values, no aliases
    brand: 'Brand(Alias)',              // semantic aliases, one mode per brand
    dimension: 'ScreenType',            // responsive overrides
  },

  // Brand names = mode names in the brand collection
  brands: ['Bayernwerk', 'LEW'],

  tokens: {
    dir: 'tokens',   // where pull writes and build reads from
  },

  outputs: {
    css: {
      outDir: 'build/css',
      prefix: '--ds',                  // CSS variable prefix
    },
    tailwind: {
      outDir: 'build/tailwind',
      version: 3,
    },
    ios: {
      outDir: 'build/ios',
      lang: 'swift',
    },
    android: {
      outDir: 'build/android',
      lang: 'compose',                 // 'xml' | 'compose'
    },
  },
})
```

## Implementation Notes

```ts
import { z } from 'zod'

const OutputTargetCSS = z.object({
  outDir: z.string(),
  prefix: z.string().default('--'),
})

const ConfigSchema = z.object({
  figma: z.object({
    fileKey: z.string().min(1),
    personalAccessToken: z.string().min(1),
  }),
  layers: z.object({
    primitives: z.string().optional(),
    brand: z.string().optional(),
    dimension: z.string().optional(),
  }).optional(),
  brands: z.array(z.string()).optional(),
  tokens: z.object({ dir: z.string().default('tokens') }).default({}),
  outputs: z.object({
    css: OutputTargetCSS.optional(),
    tailwind: z.object({ outDir: z.string(), version: z.union([z.literal(3), z.literal(4)]).default(4) }).optional(),
    ios: z.object({ outDir: z.string(), lang: z.literal('swift').default('swift') }).optional(),
    android: z.object({ outDir: z.string(), lang: z.union([z.literal('xml'), z.literal('compose')]).default('xml') }).optional(),
  }).default({}),
})

export type Config = z.infer<typeof ConfigSchema>
export const defineConfig = (config: Config) => config
```

## Implementation Comments

**2026-03-13 — Initial implementation merged.**

- **Zod not used:** The npm registry was unavailable at implementation time, so Zod
  could not be installed. Validation is done with plain TypeScript functions in
  `src/config/schema.ts` that produce the same error semantics (field-specific
  `ConfigValidationError`). Migrating to Zod later is straightforward — replace the
  `validate*` functions with `z.object(...).parse()` calls; the `Config` type stays
  the same.
- Config file is loaded via dynamic `import()` in `src/config/loader.ts`, which means
  `tsx` handles TypeScript compilation transparently at runtime.
- Types and `defineConfig` helper are exported from `src/config/index.ts` for consuming
  project autocompletion.
- Tests are in `src/config/schema.test.ts` (17 cases covering happy path + all error branches).
